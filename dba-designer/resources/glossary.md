# 术语表

> 本文件定义 dba-designer 技能中使用的专业术语和缩写，供 AI 在数据库设计时保持术语一致性。

---

## 核心术语

| 术语 | 含义 | 使用场景 |
|-----|------|---------|
| **DDL** | Data Definition Language，数据定义语言，包含 CREATE/ALTER/DROP 等 | 全流程 |
| **DML** | Data Manipulation Language，数据操纵语言，包含 INSERT/UPDATE/DELETE | 初始化数据节 |
| **全量脚本** | 包含建库→建表→索引→初始数据的完整可执行脚本，幂等设计 | Step 5 |
| **增量脚本** | 针对已有数据库结构的变更脚本（ALTER TABLE），含回滚方案 | Step 4.5 |
| **幂等** | 脚本重复执行不产生副作用（使用 IF NOT EXISTS / IF EXISTS 防护） | DDL 生成规范 |
| **逻辑外键** | 在注释中声明关联关系，由应用层维护，不使用 FOREIGN KEY 约束 | 外键处理规范 |
| **软删除** | 通过 `deleted_at` 字段标记删除，数据不物理删除 | 必备字段 |
| **覆盖索引** | 索引包含查询所需的所有字段，避免回表查询 | 索引设计 |
| **联合索引** | 包含多个列的复合索引，遵循最左前缀原则 | 索引设计 |
| **选择性** | 索引字段中不重复值的比例，越高越适合建单列索引 | 索引选型 |
| **回表** | 通过非聚簇索引找到主键后再查聚簇索引的操作，会增加 IO | 索引优化 |
| **深分页** | OFFSET 值很大的分页查询（如 LIMIT 100000, 20），性能急剧下降 | 性能问题 |
| **游标分页** | 使用上一页最后一条记录的主键作为起始条件，替代 OFFSET 分页 | 分页优化 |
| **种子数据** | 系统初始化时必须预置的枚举字典、配置项、初始账号等数据 | 初始化数据节 |
| **分区表** | 按时间/范围/哈希将一张大表物理分割，提升查询和维护效率 | 大表优化 |
| **前缀索引** | 只对字符串字段的前 N 个字符建索引，减小索引体积 | TEXT 字段处理 |

## 数据类型选择规范

| 业务场景 | 推荐类型 | 禁止类型 | 原因 |
|---------|---------|---------|------|
| 主键 ID | `BIGINT UNSIGNED` / `BIGSERIAL` | `INT`（可能溢出）、`UUID`（字符串主键索引效率差） | 分布式扩展安全 |
| 金额（分） | `BIGINT`（单位：分） | `DECIMAL`、`FLOAT`、`DOUBLE` | 浮点精度问题，违反 BR-DATA |
| 时间戳 | `DATETIME(3)` / `TIMESTAMPTZ` | `DATETIME`（无精度）、`TIMESTAMP`（2038问题） | 精确到毫秒，防并发时序问题 |
| 布尔值 | `TINYINT(1)` / `BOOLEAN` | `CHAR(1)`、`VARCHAR` | 语义清晰，存储紧凑 |
| 枚举状态 | `TINYINT` / `SMALLINT` + COMMENT | `ENUM` | ENUM 变更需 ALTER TABLE，成本极高 |
| 短字符串（<= 255） | `VARCHAR(N)` 按实际最大长度 | `VARCHAR(255)` 无差别使用 | 过长 VARCHAR 影响索引效率 |
| 长文本（不可查询） | `TEXT` / `MEDIUMTEXT` | 不建索引即可 | — |
| 长文本（可查询） | 考虑拆表或全文索引 | `TEXT` 建普通索引 | TEXT 无法建普通索引 |
| IP 地址 | `VARCHAR(45)`（兼容 IPv6） | `INT` 存储（可读性差） | 维护成本低 |
| 百分比 / 比率 | `DECIMAL(5,2)` | `FLOAT` | 精度要求明确 |
| JSON 数据 | `JSON` 类型（MySQL 5.7+ / PG） | `TEXT` | 支持 JSON 函数查询 |
| 树形结构路径 | `VARCHAR(1000)` 闭包表路径 | 递归 parent_id 深度查询 | 闭包表查询性能更好 |

## 索引类型定义

| 索引类型 | 标识前缀 | 使用场景 | 示例 |
|---------|---------|---------|------|
| 主键索引 | `pk_` | 每张表必须有，且唯一 | `pk_id` |
| 唯一索引 | `uk_` | 业务唯一约束（非主键） | `uk_user_phone` |
| 普通索引 | `idx_` | 高频查询条件字段 | `idx_order_status_created` |
| 联合索引 | `idx_` + 字段组合 | 多字段组合查询 | `idx_tenant_status_created` |
| 前缀索引 | `idx_` + 字段 + 长度 | VARCHAR 超长字段 | `idx_url_prefix` |
| 全文索引 | `ft_` | 中英文全文搜索 | `ft_content` |

## 命名规范

| 对象 | 规范 | 示例 | 禁止 |
|-----|------|------|------|
| 表名 | `snake_case`，复数名词 | `order_items` | 驼峰命名、`tbl_` 前缀 |
| 字段名 | `snake_case`，名词或名词短语 | `user_id`、`created_at` | 驼峰命名、单字母命名（除 `id`） |
| 索引名 | `{类型前缀}_{表名简写}_{字段组合}` | `idx_order_status_created` | 超过 64 字符 |
| 数据库名 | `{项目名}_db`，全小写 | `shop_db` | 含大写字母、含空格 |
| 中间表 | `{表A}_{表B}_rel` | `user_role_rel` | `{表A}_{表B}` 无后缀（可读性差） |

## 禁用词汇对照表

> 以下左列表达禁止出现在 DDL 或设计说明中，应替换为右列的规范表达。

| 禁用表达 | 替代表达 |
|---------|---------|
| "加个索引优化一下" | ❌ 必须明确：索引名称、字段列表、支持的查询接口路径 |
| "类型先用 VARCHAR" | ❌ 必须根据业务场景选择具体类型和长度，见数据类型规范 |
| "外键关联" | ✅ "逻辑外键关联（注释说明），应用层维护一致性" |
| "以后再加索引" | ❌ 禁止延后索引设计；必须在建表时一次性定义所有已知查询索引 |
| "临时用 TEXT" | ❌ 明确字段用途后选择合适类型，TEXT 不可建普通索引 |
| "枚举用 ENUM" | ✅ "用 TINYINT + COMMENT 存储枚举，详见枚举值注释格式" |
| "物理删除" | ✅ "硬删除（需在设计说明中说明不使用软删除的理由）" |
| "先建个简单的表" | ❌ 必须一次性补全所有必备字段（id/created_at/updated_at/deleted_at）和已知约束 |

## LC-002 数据库类型对照

| LC-002 值 | 数据库 | 方言关键差异 |
|----------|-------|------------|
| `MySQL` / `mysql` | MySQL 8.0 | `ENGINE=InnoDB`、`CHARSET=utf8mb4`、`AUTO_INCREMENT`、`DATETIME(3)` |
| `PostgreSQL` / `postgres` / `pg` | PostgreSQL 14+ | `BIGSERIAL`、`TIMESTAMPTZ`、`IF NOT EXISTS` 内建支持、无 ENGINE 子句 |
| `TiDB` / `tidb` | TiDB 5.x+ | 语法兼容 MySQL，但需注意 `AUTO_RANDOM` 替代 `AUTO_INCREMENT` 主键 |
| 未定义 | 默认 MySQL 8.0 | 使用 MySQL 8.0 语法生成，文件头注明"请根据实际数据库调整方言" |

## 范式术语

| 范式 | 要求 | 违反表现 | 处理方式 |
|-----|------|---------|---------|
| **1NF** | 每个字段是原子值，不可再分 | 字段存 JSON 数组、逗号分隔多值 | 拆分为独立字段或关联表 |
| **2NF** | 非主键字段完全依赖于主键（消除部分依赖） | 联合主键表中，某字段只依赖主键的一部分 | 拆表 |
| **3NF** | 非主键字段不传递依赖于主键（消除传递依赖） | 字段 A 依赖字段 B，字段 B 依赖主键 | 提取字段 B 和 A 到新表 |
| **反范式** | 适度冗余，提升查询性能 | — | 必须在设计决策记录中说明冗余字段的维护策略 |
