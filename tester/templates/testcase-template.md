# 测试用例文档模板

> 本模板用于 tester 技能阶段一产出的测试用例文档。每个被测模块独立一份文件：`doc/tester/{模块名}_测试用例.md`。

---

## 文档头部（必填，每份文档均须完整填写）

```markdown
---
文档编号：TC-YYYYMMDD-NNN（如 TC-20240601-001，同日多模块递增 NNN）
版本：v1.0.0
状态：🟡 草稿 → 🟢 已确认（评审通过后更新）
创建日期：YYYY-MM-DD
最后更新：YYYY-MM-DD
作者：{测试工程师姓名或"AI-Tester"}
关联文档：
  - 详设文档：doc/detailed/{模块}_{功能域}.md
  - 代码评审报告：doc/review/{模块名}_代码评审报告.md（若存在）
  - 项目规则：doc/detailed/项目规则.md
---
```

---

## 文档结构说明

| 章节 | 必填 | 内容 |
|------|-----|------|
| 1. 测试范围 | 是 | 被测模块名称、接口列表、测试类型分布 |
| 2. 前置条件 | 是 | 全局前置条件（如：数据库、环境、测试账号） |
| 3. 单元测试用例 | 是 | UNIT 类型用例，来自业务规则 + 代码分支 |
| 4. 集成测试用例 | 是 | INTG 类型用例，来自接口定义 |
| 5. 安全测试用例 | 按需 | SEC 类型用例，来自安全要求；无则注明「本模块无安全测试用例」 |
| 6. 性能测试用例 | 按需 | PERF 类型用例，来自性能要求；无则注明「本模块无性能测试用例」 |
| 7. 前端测试用例 | 按需 | FE 类型用例，仅 LC-FE-001 存在时填写 |
| 8. 用例统计 | 是 | 各类型用例数量汇总 |
| 变更记录 | 是 | 至少含 v1.0.0 初始版本记录 |

## 2. 全局前置条件

> 以下前置条件适用于本文档所有测试用例，各用例中不重复说明。

1. 测试环境数据库已按 `src/test/resources/test-data.sql` 初始化
2. 测试服务已启动（或 Mock 环境已就绪）
3. 已存在有效的测试账号：
   - 普通用户：`user@test.local`（密码：`Test@1234`）
   - 管理员：`admin@test.local`（密码：`Admin@1234`）
4. 外部服务依赖已使用 Mock 替代（如：第三方支付、短信服务）

## 字段填写规范

### 优先级

| 优先级 | 含义 | 典型场景 |
|-------|-----|---------|
| **P0** | 必须通过，阻塞发布 | 核心业务流程、安全漏洞、数据丢失场景 |
| **P1** | 重要，需优先修复 | 主要功能异常、接口契约覆盖 |
| **P2** | 一般，计划修复 | 边界条件、非核心错误路径 |
| **P3** | 低优先级 | UI 细节、文案偏差 |

### 测试步骤写法规范

```
✅ 正确示例：
  1. 调用 POST /api/users，请求体为 {"name": "测试用户", "email": "test@test.local"}
  2. 验证响应状态码为 201
  3. 验证响应体包含 data.id 字段（非空字符串）
  4. 查询数据库，验证 users 表中存在该记录

❌ 错误示例（过于笼统，不可操作）：
  1. 发送创建请求
  2. 验证成功
```

### 预期结果写法规范

```
✅ 正确示例（具体、可验证）：
  HTTP 状态码 = 201
  响应体：{"code": 0, "data": {"id": "非空", "name": "测试用户"}}
  数据库 users 表中存在 email = "test@test.local" 的记录

❌ 错误示例（模糊，不可验证）：
  创建成功
  返回正确结果
```

### 备注字段保留标注

| 标注 | 含义 |
|-----|-----|
| `[评审专项]` | 来自代码评审报告 P0/P1 问题的专项用例 |
| `[重点验证]` | 评审结论"有条件通过"时标注的高关注用例 |
| `[待补充]` | 详设文档对应章节缺失，待补充详设后完善该用例 |
| `[暂跳过]` | 当前阶段不执行（如性能测试需专项环境），原因已说明 |
