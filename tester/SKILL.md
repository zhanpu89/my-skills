# Tester

两阶段测试。**阶段一**：根据详设设计可评审的测试用例文档；**阶段二**：基于确认的用例生成测试代码并执行。输入：`doc/detailed/`；输出：`doc/tester/` + `src/test/`。

## 产出物

| 产出物 | 阶段 | 本质 |
|-------|-----|------|
| `{模块名}_测试用例.md` | 一 | 可评审的用例设计文档，含ID/前置/步骤/预期 |
| `src/test/` 测试代码 | 二 | 可执行的测试代码骨架 |
| `{模块名}_测试报告.md` | 二 | 标准化测试结论，含缺陷清单 |

## 懒加载原则

本技能遵守"按需加载"原则，每次只加载当前工作阶段需要的参考文件，完成后立即释放：

| 阶段 | 保持加载 | 释放 |
|------|---------|------|
| 阶段一 Step 1 | `test-patterns.md` + `testcase-template.md` | — |
| 阶段一 Step 1.5 | `testcase-template.md`（续） | `testcase-template.md`（完成后释放） |
| 阶段一 → 阶段二切换 | `glossary.md`（全流程保留） | `test-patterns.md`（已不再需要） |
| 阶段二 Step 2 | `lang-test-patterns.md`（仅对应语言） | — |
| 阶段二 Step 4 | `report-template.md` | `lang-test-patterns.md` + `report-template.md`（完成后释放） |

## 合并原则

本技能将**相同生命周期**的内容合并在一个文件中，通过 `##` 标题导航：

- **`lang-test-patterns.md`**（501行，4语言一体）：4种语言的测试代码模式合并为一个文件，按 $LC-001 路由到 `#java` / `#python` / `#nodejs` / `#go`，仅加载当前语言部分。各语言生命周期相同（阶段二 Step 2）且互斥加载，故合并在同一文件。

**分拆说明**：
- **`test-patterns.md`**（161行，仅通用内容）：原始文件 717 行中包含 557 行 Java 专属代码，与 `lang-test-patterns.md#java` 内容重复。按 Principle 1（按需加载）已将 Java 专属节移除——这些内容仅在阶段二需要，而 `test-patterns.md` 在阶段一加载。Java 测试模式请用 `lang-test-patterns.md#java`。

## 工作流

### 【阶段一：用例设计】

### Step 0：启动检测

**检测1：** 恢复模式（有`⏳`状态文档→继续）；无则检测2。
**检测2：** 阶段识别：无用例文档→阶段一；有且`src/`存在→阶段二。
**检测3：** 增量回归识别：有已有用例→修复模式（仅更新变更部分，版本号递增）。

`doc/detailed/` 无后端详设则停止。

### Step 1：文档解析与用例设计

加载 `test-patterns.md` 和 `testcase-template.md`。解析 `doc/detailed/{模块}_{功能域}.md`：

| 章节 | 提取内容 | 用例类型 |
|------|---------|---------|
| §2 业务规则 | 每条BR→正向+反向各1 | UNIT |
| §3 OpenAPI | 每接口正常/异常响应 | INTG |
| §4 伪代码 | 每个 if/else 分支 | UNIT |
| §9 性能 | RT/QPS 目标 | PERF |
| §10 安全 | 认证/越权/注入 | SEC |
| §11 测试要点 | 重点场景 | 全类型 |

额外输入：代码评审报告P0/P1→专项用例标注`[评审专项]`；全局规则BR-GLOBAL→正反向用例。

**用例ID：** `TC-{模块缩写}-{类型缩写}-{序号}`（UNIT/INTG/SEC/PERF/FE）。

**覆盖策略：** 单元=业务规则+分支+边界+异常；集成=API+业务流程+数据一致性+认证；安全/性能来自对应章节；全栈模式含前端/小程序用例。

展示设计概要等待用户确认。单模块节奏（每写完等待"继续"）。

**写入前检查：** 头部完整、ID无重复、每条含ID/标题/优先级/前置条件/步骤/预期结果、BR正反向完整、[评审专项]无遗漏。

### Step 1.5：用例修复（评审闭环）

读取评审报告或用户意见→分类处理（缺失=补充、错误=修正、重复=合并）。输出修改摘要等待确认。版本号递增（v1.0.0→v1.1.0）。

### 【阶段二：测试执行】

### Step 2：测试代码生成

前置检查：用例文档存在且已确认（非草稿）。加载 `lang-test-patterns.md`。以用例文档为唯一生成依据。每条UNIT→`@Test`/`def test_`/`func Test`，每条INTG→接口测试。Mock外部依赖。

### Step 3：测试执行

有代码时按语言执行（Java=mvnw, Python=pytest, Go=go test, Node=jest）。记录PASS/FAIL/BLOCKED/SKIP。
无代码时做静态分析，所有用例标SKIP。

### Step 3.5：缺陷记录

FAIL触发。格式：`BUG-{模块}-{序号}` + 关联用例 + 描述 + 严重程度(P0-P3)。P0立即暂停测试等待用户决策。

### Step 4：测试报告生成

加载 `report-template.md`。输出 `doc/tester/{模块名}_测试报告[_RN].md`。
多轮测试：读上轮报告+修复说明，优先回归失败用例，新增「上轮缺陷回归情况」节。

**结论标准：** ✅通过（无P0/P1，P2≤3，PASS≥90%）；⚠️有条件（无P0，PASS≥80%）；❌不通过（有P0或PASS<80%）。

**写入前检查：** 头部完整、PASS+FAIL+BLOCKED+SKIP=总用例数、P0在结论中体现、多轮含回归节。

## 全局熔断

1. `doc/detailed/` 无后端详设→停止；2. 阶段二用例文档不存在或草稿→停止；3. LC-001未知→停止；4. P0缺陷→暂停；5. 非批量授权多模块未等"继续"→停止。
